# Blockchain-Enabled Federated Learning Approach for Vehicular Networks

## Quick Facts
- arXiv ID: 2311.06372
- Source URL: https://arxiv.org/abs/2311.06372
- Reference count: 0
- Primary result: Achieved 91.92% MNIST classification accuracy in federated learning with malicious vehicles using blockchain validation

## Executive Summary
This study proposes a decentralized federated learning model for vehicular networks enhanced with blockchain technology to address data privacy and security concerns. The approach introduces a dynamic role-based system where vehicles act as workers, validators, or miners, with a validation mechanism to filter malicious updates. Using a CNN model on the MNIST dataset, the system demonstrates robustness against adversarial attacks while maintaining data privacy and integrity in a decentralized environment.

## Method Summary
The method employs a CNN model with 2D convolution layers (kernel size 2, stride 1 for convolution, stride 2 for max-pooling) trained on MNIST dataset distributed across 20 vehicles. The system implements a dynamic role-based architecture where vehicles randomly switch between worker, validator, and miner roles in each communication round. Workers perform local model training, validators evaluate updates using accuracy thresholds, and miners aggregate validated updates into blockchain blocks. The validation mechanism filters malicious updates by comparing local and validator test set accuracies, while blockchain ensures tamper-proof recording of votes and rewards.

## Key Results
- Achieved 91.92% accuracy on MNIST classification with malicious vehicles present
- Validation mechanism successfully filtered distorted model updates from malicious nodes
- Blockchain integration ensured data integrity and accountability in model aggregation

## Why This Works (Mechanism)

### Mechanism 1
The validation mechanism filters malicious updates by having validators evaluate the accuracy difference between local model test sets and their own test sets, voting based on whether accuracy drops exceed a tolerance threshold.

**Core assumption**: Small accuracy drops reliably indicate malicious or distorted updates.

Role-switching prevents persistent adversarial influence by randomly reassigning vehicles as workers, validators, or miners in each communication round.

## Architecture Onboarding

- **Component map**: Worker vehicles -> Validators -> Miners -> Blockchain ledger -> Global model

- **Critical path**:
  1. Worker trains on local data and sends update to validator
  2. Validator evaluates and votes on update
  3. Validator sends vote and reward to miner
  4. Miner aggregates votes, mines block, and broadcasts
  5. All vehicles accept block and update global model

- **Design tradeoffs**:
  - Accuracy vs. robustness: Validation threshold balances false positives and false negatives
  - Decentralization vs. latency: Role-switching and blockchain add overhead but improve security
  - Complexity vs. scalability: Dynamic roles and blockchain increase system complexity, which may impact scaling

- **Failure signatures**:
  - Accuracy drops suddenly: Possible validator compromise or ineffective validation
  - Consensus stalls: Possible miner collusion or network partitioning
  - Block propagation delays: Possible network congestion or miner shortage

- **First 3 experiments**:
  1. Run with all vehicles benign, verify baseline accuracy matches expected
  2. Introduce malicious vehicles with random updates, measure validation effectiveness
  3. Remove blockchain layer, run with centralized aggregator, compare accuracy and security resilience

## Open Questions the Paper Calls Out

### Open Question 1
- **Question**: How does the proposed model's performance scale with an increasing number of vehicles and malicious nodes?
- **Basis in paper**: [explicit] The paper acknowledges that scaling up the network may introduce additional challenges not addressed in experiments with a limited number of vehicles.
- **Why unresolved**: The current study only tested with 20 vehicles, leaving the performance and robustness in larger networks unexplored.
- **What evidence would resolve it**: Experiments with larger networks (e.g., hundreds or thousands of vehicles) to evaluate accuracy, latency, and security under varying malicious node ratios.

### Open Question 2
- **Question**: What is the impact of network delays on model updates and how can latency be mitigated?

- **Why unresolved**: The current experiments were conducted in a controlled environment without accounting for real-world network delays.
- **What evidence would resolve it**: Simulations or real-world tests incorporating network latency to assess its effect on model convergence and accuracy, along with proposed mitigation strategies.

- **Question**: How does the proposed model handle non-IID (non-independent and identically distributed) data distributions among vehicles?
- **Basis in paper**: [inferred] The paper assumes randomly distributed distinct datasets for each vehicle, but does not explore non-IID data scenarios, which are common in real-world applications.
- **Why unresolved**: The study does not provide insights into the model's performance when data distributions are skewed or heterogeneous.
- **What evidence would resolve it**: Experiments with non-IID data distributions to evaluate accuracy, convergence, and robustness under such conditions.

### Open Question 4

- **What evidence would resolve it**: Systematic experiments varying the validation threshold to analyze its impact on accuracy, computational overhead, and resilience to malicious nodes.
