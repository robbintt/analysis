# Multi-Robot Task Assignment and Path Finding for Time-Sensitive Missions with Online Task Generation

## Quick Facts

## Executive Summary
This work presents TSOTAN, a framework that integrates Multi-Robot Task Assignment (MRTA) and Multi-Agent Path Finding (MAPF) for time-sensitive missions with online task generation. TSOTAN efficiently incorporates newly generated tasks by assessing the quality of partial task reassignments and performing complete reoptimization only when the makespan exceeds a user-specified suboptimality bound. The framework uses an initial optimal MRTA solution to track a lower bound on the optimal makespan, allowing it to determine when approximate solutions are sufficient. Experiments in 2D environments demonstrate that TSOTAN produces high-quality solutions with computation times suitable for online implementation, outperforming both greedy and complete reassignment methods.

## Method Summary
TSOTAN addresses the challenge of online task generation in multi-robot systems by integrating MRTA and MAPF. The framework begins with an initial optimal MRTA solution using MILP optimization, then employs windowed path finding (RHCR-based MAPF) to generate collision-free paths. When new tasks arrive, TSOTAN attempts partial reassignment by appending tasks to existing robot assignments. If the resulting makespan exceeds a user-specified suboptimality bound (γ), the system triggers complete reassignment. A time-based lower bound on the optimal makespan is maintained to assess solution quality. The system continuously monitors for deadlocks and new tasks, replanning only when necessary to balance solution quality and computational efficiency.

## Key Results
- TSOTAN achieves makespans within 2.8% of optimal while reducing computation time by an order of magnitude compared to complete reassignment
- Compared to greedy methods, TSOTAN achieves 24.5% makespan reductions with modest increases in computation time
- The framework produces high-quality solutions with computation times suitable for online implementation in 2D environments

## Why This Works (Mechanism)

### Mechanism 1
- Claim: The system can efficiently incorporate new tasks without recomputing the entire MRTA solution by using partial reassignment when possible.
- Mechanism: When a new task arrives, the algorithm attempts a partial task assignment by adding the new task to the end of existing robot assignments. If the resulting makespan is within a user-specified suboptimality bound (γ), it accepts this solution. Only if the bound is exceeded does it trigger a complete reassignment.
- Core assumption: Partial reassignment preserves the validity of the existing task assignment and produces solutions that are often close enough to optimal.
- Evidence anchors:
  - [abstract] "TSOTAN will attempt a fast but approximate task assignment. If the approximate solution exceeds a specified suboptimality bound, it will call for a complete task reassignment"
  - [section] "Partial task assignment is a faster, approximate version of complete task assignment in which all tasks in U must be assigned without changing the previous assignment"

### Mechanism 2
- Claim: The system maintains a valid lower bound on the optimal makespan that allows it to assess solution quality.
- Mechanism: Using the initial optimal MRTA solution's makespan as a baseline, the algorithm tracks a lower bound Ω(t) = J - (t - T) where J is the initial optimal makespan, T is the time it was found, and t is the current time. This bound is valid regardless of new tasks because adding tasks cannot decrease the makespan.
- Claim: By comparing partial reassignment results to this bound, the algorithm can determine when a solution is sufficiently close to optimal.
- Mechanism: The algorithm accepts partial reassignment solutions when their makespan is within γ times the lower bound, ensuring bounded suboptimality.
- Core assumption: The tracked lower bound accurately reflects the theoretical minimum makespan achievable given the time elapsed since the optimal solution.
- Evidence anchors:
  - [section] "Knowing how path costs change by adding tasks means we can set and track bounds on the optimal makespan throughout the mission"
  - [section] "Ω(t) = J - (t - T) ∀t ≥ T is a valid lower bound on the optimal makespan at any time t regardless of the used assignment and robot paths or additional tasks generated"

### Mechanism 3
- Claim: The system integrates MRTA and MAPF while maintaining computational efficiency through windowed planning.
- Mechanism: After task assignment, a windowed MAPF algorithm (RHCR) generates collision-free paths for the current execution horizon. At the end of each window, the system checks for deadlocks or new tasks, triggering reassignment only when necessary.
- Core assumption: Windowed planning provides sufficient lookahead for most situations while keeping computation manageable.
- Evidence anchors:
  - [section] "Implementing a windowed MAPF algorithm makes the overall computation cheaper while naturally leading to planning horizons where TSOTAN can evaluate the state of the mission"
  - [section] "At the end of each path finding execution window, TSOTAN will update the lower bound Ω and assess the state of the mission"

## Architecture Onboarding

- Component map: Cost Generation -> Task Assignment -> Windowed Path Finding -> Replanning Check
- Critical path: Task Assignment → Windowed Path Finding → Replanning Check. Cost Generation is performed within Task Assignment. The replanning check can trigger Task Assignment again if conditions warrant.
- Design tradeoffs: The system trades off between computation time and solution quality through the γ parameter. Lower γ values (closer to 1) produce better solutions but require more complete reassignments and higher computation time. The windowed MAPF trades off between planning horizon and computational efficiency.
- Failure signatures: Common failure modes include: (1) Deadlock detection triggers excessive task reassignments, (2) Partial reassignment consistently exceeds the suboptimality bound requiring frequent complete reassignments, (3) The MILP solver times out on large problems, (4) Windowed path finding produces suboptimal paths due to limited horizon.
- First 3 experiments:
  1. Run TSOTAN with γ = 1 (forcing complete reassignment) and compare makespan and runtime to verify it matches the "complete method" baseline
  2. Test TSOTAN with varying γ values (1, 1.5, 2, 5) on a simple 2D environment to observe the tradeoff between makespan and computation time
  3. Simulate task generation patterns (bursty vs. steady) to evaluate how different patterns affect the frequency of partial vs. complete reassignments

## Limitations
- The framework's effectiveness depends heavily on the quality of the initial optimal solution, which serves as the baseline for tracking makespan bounds
- In highly dynamic environments with frequent task generation, the system may need to perform complete reassignments more often, potentially negating the computational efficiency gains
- The paper focuses on 2D grid environments, and performance in continuous 3D spaces or unstructured environments remains untested

## Confidence

- **High Confidence**: The core mechanism of partial task reassignment with bounded suboptimality is well-supported by the experimental results showing 2.8% makespan degradation compared to optimal while achieving 10x speedup.
- **Medium Confidence**: The integration of MRTA and MAPF through windowed planning is conceptually sound but depends on the RHCR algorithm's effectiveness, which is not fully detailed in the paper.
- **Medium Confidence**: The time-based lower bound tracking mechanism appears valid but may become less reliable for very long missions where the theoretical minimum makespan grows significantly.

## Next Checks

1. **Scalability testing**: Evaluate TSOTAN's performance on larger problem instances (e.g., 10+ robots, 50+ tasks) to assess the practical limits of the MILP-based task assignment component and the frequency of complete reassignments.

2. **Dynamic environment stress test**: Create scenarios with high task generation rates (e.g., 50% probability per timestep) to determine how often partial reassignment fails the suboptimality bound, forcing complete reassignment and reducing efficiency gains.

3. **Alternative MAPF integration**: Replace RHCR with other MAPF algorithms (e.g., ECBS, Lazy CBS) in the windowed planning component to assess whether different path finding approaches affect the overall system's performance and deadlock handling capabilities.
