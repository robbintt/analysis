# Conflict-Aware Active Automata Learning (Extended Version)

## Quick Facts
- arXiv ID: 2308.14781
- Source URL: https://arxiv.org/abs/2308.14781
- Reference count: 40
- Primary result: C3AL achieves 95.5% success rate vs MAT's 79.5% in noisy active automata learning

## Executive Summary
C3AL introduces a framework to handle conflicting information during active automata learning by separating the learner's data structure from an external observation tree managed by a Reviser agent. This allows efficient conflict resolution without restarting the learner, enabling learning in noisy or mutating environments. Experiments across 30+ realistic targets with various noise levels show C3AL achieves a 95.5% success rate versus MAT's 79.5%, particularly excelling on complex systems with high noise where MAT fails. C3AL reduces the number of system tests needed by 6% on average while maintaining correctness.

## Method Summary
The paper proposes C3AL as an alternative to the MAT framework for active automata learning. C3AL introduces a Reviser agent that maintains an external observation tree storing all non-conflicting observations. When conflicts arise, the Reviser updates the tree and signals the learner to prune and rebuild from the conflict-free point. The framework supports two update strategies: Most Recent (committing the latest observation) and Most Frequent (committing the most common observation). Experiments compare C3AL against MAT across 36 real-world Mealy machine benchmarks with three noise types (input/output/both) at three levels (0.01%, 0.05%, 0.1%) using four learning algorithms (L#, TTT, KV, L*).

## Key Results
- C3AL achieves 95.5% success rate versus MAT's 79.5% across all benchmarks and noise levels
- C3AL reduces system tests by 6% on average through caching observations
- C3AL successfully learns complex systems with high noise where MAT fails completely

## Why This Works (Mechanism)

### Mechanism 1
- Claim: C3AL handles conflicts by storing observations in an external observation tree managed by a Reviser agent, enabling efficient conflict resolution without restarting the learner.
- Mechanism: The Reviser maintains an observation tree that caches all non-conflicting observations. When a conflict arises, the Reviser updates the tree to remove inconsistent information while preserving valid observations. The learner then prunes its internal state to a conflict-free point and rebuilds from the observation tree without requiring new tests for previously observed behaviors.
- Core assumption: Separating observation storage from the learner's internal data structure allows efficient updates and conflict resolution without losing valid information.
- Evidence anchors:
  - [abstract]: "C3AL introduces a framework to handle conflicting information during active automata learning by separating the learner's data structure from an external observation tree managed by a Reviser agent."
  - [section 3.2]: "The Reviser alone interacts with the SUL by means of tests meant to expand its observation tree. Crucially, C3AL is less abstract than MAT, representing directly the objects and challenges of practical active learning"
  - [corpus]: Weak - no direct corpus evidence for this specific mechanism, though related conflict-aware frameworks exist.
- Break condition: If the Reviser's observation tree becomes too large or complex to manage efficiently, or if conflict resolution strategies fail to maintain a coherent view of the system.

### Mechanism 2
- Claim: C3AL reduces the number of system tests needed by reusing stored observations through the observation tree, achieving a 6% average reduction in test use.
- Mechanism: When the learner queries the Reviser, the Reviser first checks its observation tree. If the answer exists, it returns the cached result without testing the system. Only when information is missing does the Reviser test the system and update the tree. This caching eliminates redundant tests for previously observed behaviors.
- Core assumption: The observation tree can be efficiently queried and maintained to provide cached answers for membership and equivalence queries.
- Evidence anchors:
  - [abstract]: "C3AL reduces the number of system tests needed by 6% on average while maintaining correctness."
  - [section 3.3]: "Membership queries can be defined, for i∈ Σ∗, as MQ(i) = READ T (i). When T does not have the answer to this particular query, READ T sends it through to the SUL (with the call to System) and the result is applied in T."
  - [section 4.2]: "C3AL provides an average reduction of 6% in test use, when compared to MAT."
- Break condition: If the observation tree grows too large to maintain efficient lookup times, or if the cost of maintaining the tree exceeds the savings from reduced system tests.

### Mechanism 3
- Claim: C3AL's update strategies (Most Recent and Most Frequent) handle different types of conflicts appropriately, allowing successful learning in noisy environments where MAT fails.
- Mechanism: When conflicts arise, C3AL can either commit the most recent observation (for mutating systems) or the most frequent observation (for noisy systems). This flexibility allows C3AL to adapt to different types of inconsistencies in the observation data.
- Core assumption: Different conflict types require different resolution strategies, and C3AL's flexible approach can handle both noise and system mutations effectively.
- Evidence anchors:
  - [section 3.2]: "We propose two ways to resolving conflicts in C3AL: 1 Most Recent: When a conflict is identified, the most recently observed query information is committed to the observation tree, and the previous one suppressed, if needed."
  - [section 4.2]: "C3AL manages to stay consistently reliable in the face of these large alphabets" and "C3AL had an overall success rate of 95.5% compared to MAT's 79.5% success rate."
  - [corpus]: Weak - no direct corpus evidence for these specific update strategies, though related conflict resolution exists.
- Break condition: If the update strategy choice is incorrect for the type of conflict present, or if neither strategy adequately resolves the conflicts in highly complex scenarios.

## Architecture Onboarding

- Component map:
  - Learner -> Reviser -> System Under Learning (SUL)
  - Learner implements learning algorithms (L*, KV, TTT, L#)
  - Reviser manages observation tree and handles conflicts
  - SUL is the actual system being learned
  - Observation Tree stores valid observations externally
  - Testing Algorithms (Wp-method, Hybrid-ADS) for equivalence testing

- Critical path:
  1. Learner sends membership/equivalence query to Reviser
  2. Reviser checks observation tree for cached answer
  3. If missing, Reviser tests SUL and updates observation tree
  4. If conflict detected during update, Reviser sends Prune signal to Learner
  5. Learner restarts from conflict-free state using observation tree
  6. Learner continues building hypothesis

- Design tradeoffs:
  - Memory vs. Efficiency: Larger observation trees provide better caching but consume more memory
  - Update Strategy: Most Recent favors recent behavior (good for mutations) vs. Most Frequent favors stable behavior (good for noise)
  - Query Complexity: C3AL reduces system tests but may increase Reviser complexity

- Failure signatures:
  - Learner repeatedly receiving Prune signals indicates frequent conflicts
  - Observation tree becoming too large or slow to query
  - Success rates dropping despite conflict handling mechanisms
  - Memory exhaustion from maintaining large observation trees

- First 3 experiments:
  1. Implement a simple Mealy machine learner (e.g., L*) with C3AL framework using a basic observation tree and Most Recent update strategy
  2. Test on a small deterministic system with no noise to verify basic functionality
  3. Introduce controlled noise and verify that C3AL successfully handles conflicts while MAT fails

## Limitations

- The effectiveness of C3AL depends heavily on implementation details of the observation tree and conflict resolution strategies
- Experiments use synthetic noise injection rather than naturally occurring noise patterns
- Scalability to very large state spaces and complex systems is not thoroughly addressed

## Confidence

**High Confidence**: C3AL's basic framework of separating learner from observation tree management, the 95.5% vs 79.5% success rate improvement over MAT, the 6% reduction in system tests achieved through caching

**Medium Confidence**: The effectiveness of specific update strategies (Most Recent vs Most Frequent), performance across all 36 benchmarks at different noise levels, the timeout behavior and its impact on results

**Low Confidence**: Real-world applicability beyond synthetic noise models, scalability to systems with very large state spaces or alphabets, performance in environments with both noise and system mutations simultaneously
