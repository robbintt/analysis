# Action Language BC+

## Quick Facts
- arXiv ID: 2506.18044
- Source URL: https://arxiv.org/abs/2506.18044

## Executive Summary
This paper introduces BC+, an action language that integrates the best features of existing action languages (B, C, C+, BC) with modern Answer Set Programming (ASP) constructs. BC+ is defined as a high-level notation for propositional formulas under stable model semantics, enabling direct use of choice rules, aggregates, and abstract constraint atoms. The language's semantics are defined through translation to propositional formulas, allowing computational methods from ASP solvers to be readily applied. BC+ demonstrates greater expressiveness than its predecessors and provides a more convenient way to represent concepts like defeasible causal laws and action attributes.

## Method Summary
BC+ is formally defined as a high-level notation for propositional formulas under stable model semantics. The language unifies features from existing action languages (B, C, C+, BC) with modern ASP constructs such as choice rules, aggregates, and abstract constraint atoms. The semantics of BC+ are defined by translating descriptions into propositional formulas, which allows computational methods from ASP solvers to be directly applied. A system implementation extending CPLUS 2ASP is presented, and the language's advantages are demonstrated through examples including a blocks world formalization that leverages aggregates.

## Key Results
- BC+ unifies features from action languages B, C, C+, and BC
- BC+ enables direct use of ASP constructs like choice rules, aggregates, and abstract constraint atoms
- A system implementation extending CPLUS 2ASP demonstrates practical application
- Examples show BC+'s advantages, including convenient representation of defeasible causal laws

## Why This Works (Mechanism)
BC+ works by defining a high-level notation for propositional formulas under stable model semantics. This allows the language to directly incorporate modern ASP constructs while maintaining compatibility with existing action language features. The translation mechanism from BC+ descriptions to propositional formulas enables the use of established computational methods from ASP solvers. By unifying different action language paradigms, BC+ provides a more expressive framework that can represent complex scenarios more naturally than its predecessors.

## Architecture Onboarding

**Component Map:**
User Description -> BC+ Translator -> Propositional Formulas -> ASP Solver -> Stable Models -> Action Results

**Critical Path:**
1. User provides BC+ description
2. BC+ Translator converts to propositional formulas
3. ASP Solver computes stable models
4. Results interpreted as action outcomes

**Design Tradeoffs:**
- Expressiveness vs. computational complexity
- High-level notation vs. direct implementation
- Unification of paradigms vs. potential learning curve

**Failure Signatures:**
- Non-terminating ASP solver (computational complexity issues)
- Unexpected stable models (translation errors or ambiguous descriptions)
- Solver timeout (scalability limitations)
