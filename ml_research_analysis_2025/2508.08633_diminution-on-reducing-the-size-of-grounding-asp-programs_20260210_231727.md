# Diminution: On Reducing the Size of Grounding ASP Programs

## Quick Facts
- arXiv ID: 2508.08633
- Source URL: https://arxiv.org/abs/2508.08633
- Authors: HuanYu Yang; Fengming Zhu; YangFan Wu; Jianmin Ji
- Reference count: 40
- Key outcome: Introduces "diminution" technique reducing grounding size by up to 85%, improving ASP performance

## Executive Summary
This paper addresses the grounding bottleneck in Answer Set Programming (ASP) where large Herbrand universes lead to inefficient grounding and solving. The authors introduce "diminution," a technique that restricts grounding to a selected subset of constants, reducing the size of the grounded program. They formally define admissible and safe diminutions, analyze their properties, and develop a heuristic approach for selecting effective subsets. The method integrates seamlessly with existing ASP solvers via domain predicates. Experimental results on five benchmarks demonstrate significant performance improvements: up to 70% reduction in grounding time and 85% decrease in grounding file size, showcasing diminution as a robust approach to alleviate the grounding bottleneck in ASP.

## Method Summary
The diminution technique works by restricting the grounding process to a carefully selected subset of constants from the Herbrand universe. The authors formalize this approach by defining admissible diminutions (which preserve all answer sets) and safe diminutions (which preserve all answer sets without introducing spurious ones). They develop a heuristic method for selecting optimal constant subsets based on dependency analysis and relevance scoring. The technique is implemented through domain predicates that guide the grounding process, allowing integration with existing ASP solvers without requiring modifications to the underlying solving algorithms.

## Key Results
- Up to 70% reduction in grounding time across five benchmarks
- 85% decrease in grounding file size

## Why This Works (Mechanism)
The diminution approach works by exploiting the observation that not all constants in the Herbrand universe are necessary for computing answer sets. By identifying and removing irrelevant constants during the grounding phase, the technique dramatically reduces the search space size. The key insight is that grounding only the relevant subset of constants preserves all valid answer sets while eliminating the computational overhead of processing unnecessary combinations. The heuristic selection process ensures that the chosen subset maintains completeness while maximizing reduction in grounding complexity.

## Foundational Learning
**Herbrand Universe**: The set of all ground terms constructible from constants and function symbols in a logic program. Understanding this concept is essential for grasping why grounding can become computationally expensive with many constants.

**Grounding Bottleneck**: The computational challenge where the grounding phase generates an exponentially large number of ground rules from a compact non-ground program. This is the primary problem that diminution addresses.

**Domain Predicates**: Special predicates used to guide the grounding process by restricting which constants are considered during instantiation. These serve as the integration mechanism for diminution in existing ASP solvers.

**Answer Set Preservation**: The property that a diminution technique must maintain all valid answer sets of the original program. This is critical for ensuring correctness of the optimization.

**Safe vs Admissible Diminution**: Safe diminutions preserve all answer sets without introducing spurious ones, while admissible diminutions only guarantee preservation of existing answer sets. Understanding this distinction is crucial for evaluating the technique's guarantees.

## Architecture Onboarding

**Component Map**: ASP Program → Dependency Analysis → Constant Relevance Scoring → Subset Selection → Domain Predicates → Grounded Program → Solver

**Critical Path**: The most performance-critical sequence is ASP Program → Dependency Analysis → Subset Selection, as these steps directly determine the effectiveness of grounding reduction. Errors in this path can lead to either insufficient reduction or loss of answer sets.

**Design Tradeoffs**: The main tradeoff is between reduction size and answer set preservation. Larger constant subsets provide stronger guarantees but offer less performance improvement. The heuristic must balance aggressive reduction with completeness preservation.

**Failure Signatures**: 
- Insufficient reduction (subset too large) → minimal performance gains
- Over-reduction (subset too small) → missing answer sets or solver errors
- Poor heuristic selection → suboptimal subsets that miss opportunities for reduction
- Integration issues with domain predicates → solver incompatibility or incorrect grounding
