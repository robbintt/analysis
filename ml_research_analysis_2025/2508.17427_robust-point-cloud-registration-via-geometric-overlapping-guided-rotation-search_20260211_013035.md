# Robust Point Cloud Registration via Geometric Overlapping Guided Rotation Search

## Quick Facts
- **arXiv ID:** 2508.17427
- **Source URL:** https://arxiv.org/abs/2508.17427
- **Reference count:** 40
- **Primary result:** Achieves 89.46% recall on 3DMatch (FPFH), 50.59% on 3DLoMatch, and 100% on KITTI while maintaining polynomial time complexity

## Executive Summary
This paper addresses the challenging problem of point cloud registration under high outlier ratios by proposing a geometric maximum overlapping registration (GMOR) framework. The method decomposes the 6-DoF rigid transformation using Chasles' theorem into a translation along a rotation axis and a 2D rigid transformation, then performs rotation-only branch-and-bound search. By formulating residual parameters as range maximum query problems, the approach achieves superior accuracy and efficiency compared to state-of-the-art methods while maintaining linear space complexity.

## Method Summary
The GMOR method performs robust point cloud registration by first computing weighted correspondences using feature descriptors (FPFH, FCGF, or Predator). It then decomposes the rigid transformation via Chasles' theorem into a screw motion, searching for the optimal rotation axis using a 2D branch-and-bound approach with interval stabbing for axial translation estimation. The rotation angle is subsequently searched with a 1D branch-and-bound, while the rotation center is determined through a sweep line algorithm solving the Klee's rectangle problem. The pipeline concludes with Iteratively Reweighted Least Squares (IRLS) refinement.

## Key Results
- Achieves 89.46% recall on 3DMatch dataset (FPFH descriptor), outperforming baselines like TEASER++ (84.26%) and RPM-Net (85.41%)
- Demonstrates 50.59% recall on challenging 3DLoMatch dataset with low overlap
- Shows 100% recall on KITTI dataset while maintaining polynomial time complexity O(N log N) and linear space complexity O(N)
- Maintains superior performance across varying outlier ratios, with runtime advantages over graph-based methods in typical scenarios

## Why This Works (Mechanism)

### Mechanism 1: Dimensionality Reduction via Chasles' Decomposition
The method reduces the 6-DoF search space complexity by decomposing rigid transformation into a translation along a rotation axis and a 2D rigid transformation, rather than searching $SE(3)$ directly. This restructuring enables two rotation-only BnB stages: first searching the 3D rotation axis direction, then the 1D rotation angle, while treating the translation along the axis and the 2D rotation center as residuals solvable via geometric queries. The core assumption is that the input transformation is strictly rigid, and the optimal solution lies within the parameterized rotation space (Hemisphere).

### Mechanism 2: 1D Range Maximum Query (RMQ) for Axis Translation
Estimating the translation along a specific rotation axis is formulated as a weighted interval stabbing problem (1D RMQ), which provides a tight upper bound for BnB pruning. Correspondences are projected onto the candidate rotation axis, and the "inlier" condition for translation forms a 1D interval. The algorithm computes the maximum weighted overlap of these intervals to determine the optimal translation $d$ and a bound for the current BnB branch. This replaces expensive spatial compatibility graph construction with a sort-and-scan operation. The core assumption is that inliers constrain to a tight interval along the correct axis, while outliers distribute randomly or broadly.

### Mechanism 3: 2D RMQ (Klee's Rectangle Problem) for Rotation Center
The residual 2D registration (finding the rotation center $c$ for a given axis and angle) is solved deterministically by finding the point of maximum overlap among axis-aligned rectangles. For a fixed rotation angle, each correspondence constrains the rotation center to a specific geometric region (approximated as a rectangle). Finding the best rotation center equates to finding the 2D point covered by the most rectangles. This is solved via a sweep line algorithm with a segment tree in $O(N \log N)$, avoiding a nested 2D translation BnB search. The core assumption is that the rotation center lies within the intersection of the true inliers' geometric constraints; the angle discretization is sufficient to capture the true rotation.

  - **Why needed here:** The core logic relies on efficiently pruning the rotation search space. You must understand how "upper bounds" derived from the RMQ steps allow the algorithm to skip vast regions of the rotation sphere.
  - **Quick check question:** Why does finding the *maximum* overlap of intervals provide a valid *upper bound* for a specific rotation axis branch?

## Architecture Onboarding

- **Component map:** Preprocessing (Weighted Feature Matching) -> Stage I (Axis Search: Cube-mapping → 2D BnB → 1D Interval Stabbing) -> Stage II (Angle Search: 1D BnB → 2D Sweep Line/Segment Tree) -> Refinement (IRLS)

- **Critical path:** The transition from Stage I to Stage II (Top-$k$ selection). If the true rotation axis is pruned or ranked below $k$ in Stage I due to loose bounds, Stage II cannot recover the correct pose.

- **Design tradeoffs:**
  - **Top-$k$ value:** Higher $k$ increases robustness to local optima in Stage I but linearly increases Stage II computation time
  - **Branch width ($\epsilon$):** Smaller $\epsilon$ increases precision but exponentially increases BnB nodes
  - **Cube Mapping vs. Spherical:** Cube mapping provides more uniform parameterization than spherical coordinates, preventing distortion near poles

- **Failure signatures:**
  - **Translation-only degeneracy:** Fails when rotation angle is near zero; must trigger the specific "Translation-only" logic in Section VI-C
  - **Memory explosion:** Unlike graph methods, this scales linearly ($O(N)$). If memory spikes, check interval sorting buffers
  - **Small Overlap:** Performance drops on 3DLoMatch (approx. 50% Recall vs 89% on 3DMatch), indicating sensitivity to low inlier ratios

- **First 3 experiments:**
  1. **Unit Test RMQ Solvers:** Generate synthetic intervals/rectangles with known ground truth overlaps and verify the `IntervalStabbing` and `SweepLine` functions return the correct maximum weights
  2. **Parameter Sweep ($k$ vs. $\rho$):** Run ablation on 3DMatch to find the optimal $k$ (candidate axes) and convergence ratio $\rho$ that balances speed vs. recall
  3. **Degeneracy Test:** Feed point clouds with pure translation (zero rotation) and verify the code correctly switches to the approximation in Equation 22-24 rather than diverging

## Open Questions the Paper Calls Out

- **Open Question 1:** Can the selection of "top-k" candidate rotation axes in Stage I be formally constrained to guarantee that the global optimum is never pruned before Stage II?
- **Open Question 2:** How does the proposed method perform in theoretical worst-case scenarios where BnB branch pruning is minimized, specifically compared to graph-based methods?
- **Open Question 3:** How robust is the translation-only approximation when the rotation angle approaches zero, given that the noise threshold $\xi_c(\theta)$ and rotation center $C$ theoretically diverge?
- **Open Question 4:** Is the superior performance of the 3+1 decomposition strategy over 1+3 intrinsic to the algorithm, or dependent on the outlier distributions in the KITTI dataset?

## Limitations

- Performance degrades significantly on 3DLoMatch dataset (50.59% recall vs 89.46% on 3DMatch), indicating sensitivity to low overlap ratios
- The method's effectiveness depends heavily on the quality of pre-computed correspondences from feature descriptors
- Worst-case BnB complexity can approach $O(1/\epsilon^2)$, potentially narrowing the runtime advantage over graph-based methods
- The translation-only approximation for near-zero rotation angles introduces approximation error without rigorous boundary characterization

## Confidence

- **High:** Chasles' theorem decomposition, branch-and-bound framework, theoretical complexity analysis
- **Medium:** Experimental results on benchmark datasets, runtime comparisons with baselines
- **Low:** Claims about RMQ solvers' robustness to noise, exact memory usage in practice
