# fCrit: A Visual Explanation System for Furniture Design Creative Support

## Quick Facts
- arXiv ID: 2508.12416
- Source URL: https://arxiv.org/abs/2508.12416
- Reference count: 25
- Key outcome: None (prototype with planned user study)

## Executive Summary
fCrit is a dialogue-based furniture design critique system that combines multi-agent AI architecture with a structured knowledge base of visual concepts and patterns. The system provides adaptive, explainable feedback tailored to user awareness levels through on-demand knowledge retrieval and reflective scaffolding techniques. Designed to support creative exploration while maintaining domain-grounded responses, fCrit aims to help designers articulate tacit knowledge and deepen their formal understanding of design elements.

## Method Summary
fCrit employs a five-agent architecture orchestrated through n8n workflow automation, using different Claude models for specific tasks: Claude 3.5 Haiku for input processing, Claude 3 Haiku for knowledge retrieval, and Claude 3.7 Sonnet for dialogue generation. The system features on-demand querying of a vector-based knowledge store containing formal design concepts and patterns with awareness-level terminology, confidence scoring for concept extraction, and reflective scaffolding through rephrasing, generative questioning, and visual analogies.

## Key Results
- Functional prototype demonstrating multi-agent architecture for furniture design critique
- Knowledge base encoding of visual concepts with formal definitions, perceptual effects, applications, examples, and awareness-specific terminology
- Three-tier scaffolding techniques (rephrasing, generative questions, visual analogies) grounded in reflection-in-action and reflective learning frameworks

## Why This Works (Mechanism)

### Mechanism 1
Multi-agent orchestration with on-demand knowledge retrieval reduces cognitive overload while maintaining domain-grounded responses. Five specialized agents operate in a three-tier workflow, querying the vector-based knowledge store only when relevant concepts are detected, enabling focused, context-sensitive support rather than pre-emptively loading all possible design knowledge.

### Mechanism 2
Hierarchical knowledge embeddings with awareness-level tailoring enable adaptive explanation without reformulating underlying design principles. The knowledge base encodes visual concepts with formal definitions, perceptual effects, applications, examples, and "Formal Awareness Levels" (Novice, Intermediate, Expert), allowing the Design Concept Mapper to tailor outputs to detected user awareness levels.

### Mechanism 3
Reflective scaffolding through rephrasing, generative questioning, and visual analogies promotes articulation of tacit knowledge. The Dialogue Agent synthesizes insights from prior agents and applies three techniques grounded in Schön's reflection-in-action and Moon's reflective learning frameworks to help users articulate their design intuitions.

## Foundational Learning

- **Formal Critique Methodology** (visual elements: line, shape, form; visual patterns: balance, contrast, unity)
  - Why needed here: The knowledge base and all agent outputs presuppose familiarity with formal design vocabulary. Without this, the Design Concept Mapper's extractions will not resonate.
  - Quick check question: Can you articulate the difference between "formal elements" and "visual patterns" in design analysis?

- **Reflective Learning Theory** (Schön's reflection-in-action, Moon's reflective learning)
  - Why needed here: The Dialogue Agent's scaffolding techniques are theoretically grounded in these frameworks; understanding them clarifies why the system asks generative questions rather than providing direct answers.
  - Quick check question: What is the difference between "reflection-in-action" and "reflection-on-action," and why might the former be more relevant to design critique?

- **Mixed-Initiative Interaction**
  - Why needed here: fCrit frames human-AI dialogue as co-constructive rather than unidirectional; the system takes initiative when it enhances the user's trajectory, not arbitrarily.
  - Quick check question: In a mixed-initiative system, what criteria determine when the AI should take initiative versus defer to the user?

## Architecture Onboarding

- **Component map**: User input → Command Hub → (Design Concept Mapper + Pattern Recognition Engine) → Vector Store → Etiquette Classifier → Dialogue Agent → Output
- **Critical path**: User submits input → Command Hub routes to relevant agents → Design Concept Mapper/Pattern Recognition Engine query Vector Store if relevant → Etiquette Classifier determines tone/length → Dialogue Agent synthesizes and generates scaffolded response → Confidence scores logged
- **Design tradeoffs**: On-demand vs. pre-loaded knowledge (reduces overload but introduces latency), sequential vs. parallel agent activation (maintains focus but may miss synergies), awareness-level tailoring vs. universal language (improves accessibility but risks misclassification)
- **Failure signatures**: Low confidence scores from Concept Mapper/Pattern Recognition Engine (knowledge base coverage gaps), user repeatedly ignores generative questions (scaffolding mismatch), response length inappropriate despite Etiquette Classifier (classifier calibration issue)
- **First 3 experiments**: 
  1. Conduct confidence score audit across 50+ diverse design queries to identify knowledge base coverage gaps
  2. Perform think-aloud sessions with 3 novice and 3 expert designers to detect awareness-level mismatch patterns
  3. Test whether scaffolded generative questions measurably increase users' use of formal terminology compared to direct feedback conditions
