# Action Language BC+

## Quick Facts
- arXiv ID: 2506.18044
- Source URL: https://arxiv.org/abs/2506.18044
- Authors: Joseph Babb; Joohyung Lee
- Reference count: 20
- Introduces BC+, an action language unifying features of existing action languages (B, C, C+, BC) with modern ASP constructs

## Executive Summary
This paper introduces BC+, an action language that unifies features from existing action languages (B, C, C+, BC) with modern ASP constructs like choice rules, aggregates, and abstract constraint atoms. BC+ is defined as a high-level notation of propositional formulas under stable model semantics, enabling direct use of ASP constructs. The language's semantics are defined through translation to propositional formulas, allowing computational methods from ASP solvers to be readily applied.

## Method Summary
BC+ is presented as an extension of the action language C+ that incorporates modern ASP constructs. The language is formally defined through translation to propositional formulas under stable model semantics. The semantics of BC+ descriptions are defined by translating them into propositional formulas, which can then be processed using existing ASP solvers. The paper also presents a system implementation that extends CPLUS 2ASP, demonstrating the practical application of BC+ through examples including a blocks world formalization that leverages aggregates.

## Key Results
- BC+ unifies features from existing action languages (B, C, C+, BC) with modern ASP constructs
- The language allows direct use of choice rules, aggregates, and abstract constraint atoms
- BC+ is more expressive than its predecessors and enables convenient representation of concepts like defeasible causal laws and action attributes
- The blocks world formalization demonstrates BC+'s advantages through aggregate usage

## Why This Works (Mechanism)
BC+ works by serving as a high-level notation for propositional formulas under stable model semantics. This design allows direct incorporation of modern ASP constructs into action language descriptions. The translation mechanism from BC+ descriptions to propositional formulas enables computational methods from ASP solvers to be readily applied, providing both expressiveness and practical implementation capabilities.

## Architecture Onboarding

**Component Map:**
BC+ Description -> Translation Engine -> Propositional Formulas -> ASP Solver -> Stable Models

**Critical Path:**
BC+ description input → syntactic translation to propositional formulas → stable model computation → answer set generation

**Design Tradeoffs:**
- High-level expressiveness vs. computational complexity
- Direct ASP construct usage vs. potential loss of action-specific optimizations
- Unified framework vs. specialized handling of specific action language features

**Failure Signatures:**
- Inconsistent translations producing no stable models
- Semantic mismatches between BC+ descriptions and target propositional formulas
- Performance degradation with complex aggregate expressions
- Incorrect handling of defeasible causal laws during translation

**3 First Experiments:**
1. Translate simple BC+ descriptions containing choice rules and verify stable model output
2. Compare BC+ blocks world formalization with traditional C+ implementation using aggregates
3. Test defeasible causal law representation in BC+ and validate against expected reasoning outcomes
